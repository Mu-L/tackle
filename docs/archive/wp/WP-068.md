## 目标

基于诊断报告 `docs/reports/2026-05-25_batch-execution-stall-diagnosis.md` 的分析结果，实施 3 个方案以解决批量执行 skill 长时间不操作后停止运行的问题。

## 问题分析

### 根因

agent-dispatcher 的监控循环每轮迭代消耗大量上下文 token（TaskList、Write、SendMessage、Read），长时间运行后触发 Claude Code 的 auto-compact（上下文压缩），导致关键状态变量丢失：

| 状态变量 | 用途 | 丢失后果 |
|----------|------|----------|
| `teamee_map` | task_id → teamee_name 映射 | 无法追踪运行中的 teammate |
| `wp_assignments` | 角色和记忆预计算结果 | 无法创建新 teammate |
| `team_name` | 团队名称 | 无法发送消息 |
| `processed_action_ids` | 已处理的 daemon 指令 | 重复处理指令 |
| `loop_iteration`、`start_time` | 循环计数 | 计时失效 |

### 三方案概览

| 方案 | 优先级 | 效果 | 实施内容 |
|------|--------|------|----------|
| 方案 1：状态持久化 | P0 | 根治 | 将关键状态写入文件，压缩后可恢复 |
| 方案 2：Watchdog 超时 | P1 | 缓解 | 增大超时阈值，防止误判 |
| 方案 3：批量大小控制 | P2 | 缓解 | 限制单批任务数，降低上下文消耗 |

## 实施计划

### WP-068-1-impl: 状态持久化 + 恢复协议 + 批量控制

**核心修改文件**：`plugins/core/skill-agent-dispatcher/skill.md`

#### Step 1：增加状态文件初始化（监控循环前）

在监控循环开始前（约第 285 行），增加状态文件写入逻辑：

```
state_file = ".claude-daemon/dispatcher-state.json"

initial_state = {
    "team_name": "{team_name}",
    "teamee_map": {},
    "wp_assignments": {task_id: {role_id, wp_doc_path, role_prompt_summary}},
    "start_time": now_iso8601(),
    "loop_iteration": 0,
    "processed_action_ids": [],
    "total_tasks": len(tasks),
    "status": "monitoring",
    "max_batch_size": 5,
    "current_batch": [],
    "pending_batches": []
}
Write(state_file, json_dumps(initial_state))
```

#### Step 2：修改监控循环 — 每轮开头恢复状态

在 Phase A 之前，增加从文件恢复状态的步骤：

```
state = json_loads(Read(state_file))
team_name = state.team_name
teamee_map = state.teamee_map
wp_assignments = state.wp_assignments
start_time = state.start_time
loop_iteration = state.loop_iteration
processed_action_ids = state.processed_action_ids
max_batch_size = state.max_batch_size
current_batch = state.current_batch
pending_batches = state.pending_batches
```

如果文件不存在或 status == "completed"，说明已结束或首次运行，正常处理。

#### Step 3：修改监控循环 — 状态变更后写回

在以下位置增加状态文件更新：

- Phase B（销毁 teamee 后）：更新 teamee_map
- Phase C（创建 teamee 后）：更新 teamee_map
- Phase D（处理 daemon 指令后）：更新 processed_action_ids
- 每轮循环末尾：更新 loop_iteration、status

#### Step 4：增加上下文恢复协议章节

在 skill.md 中新增"上下文恢复协议"章节：

```
## 上下文恢复协议

如果监控循环因上下文压缩而中断，执行以下步骤恢复：

1. Read(".claude-daemon/dispatcher-state.json")
2. 从文件恢复所有状态变量
3. 检查 team 配置是否仍然有效（Read team config）
4. 恢复 teamee_map 中记录的活跃 teamee
5. 继续监控循环
```

#### Step 5：调整 Watchdog 超时配置（方案 2）

修改 `.claude/watchdog/daemon-config.template.json`：
- `session_stalled_min`: 10 → 30
- `task_stalled_min`: 15 → 20

#### Step 6：批量大小控制参数（方案 3 代码化）

在 skill.md 中增加批量大小控制逻辑：

- 新增参数：`max_batch_size`（默认值 5），用户可通过配置覆盖
- 在任务分配阶段（Phase C 之前），检查待执行任务数是否超过 max_batch_size
- 如果超过，仅取前 max_batch_size 个任务进入本轮监控循环
- 剩余任务写入 `pending_batches` 字段
- 当前批次全部完成后，自动从 `pending_batches` 加载下一批继续执行（而非退出）
- 每批完成后更新状态文件，保存进度

**辅助修改文件**：`.claude/watchdog/daemon-config.template.json`

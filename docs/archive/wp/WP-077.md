### 2. 逐项技术可行性分析

#### A 类 — 核心能力缺失

**WP-068(→WP-078): 外部插件加载机制** — 方向正确，方案需补充

代码验证：
- `harness-build.js._resolvePluginDir()` (L399-409) **已支持绝对路径**
- `plugin-loader.js._loadPlugin()` (L425-480) **确实硬编码** `path.join(registryDir, 'core', source)`
- 两个文件的路径解析逻辑**不一致**

问题：
1. 报告未发现 `harness-build.js` 已支持绝对路径，方案中对 `_resolvePluginDir()` 的修改是多余的
2. `"source": "npm:tackle-plugin-xxx"` 方案缺少关键细节：
   - 全局安装时 npm 包的 resolve 路径不同（`require.resolve()` vs `node_modules`）
   - 未考虑插件间的 peer dependency 冲突
   - 未考虑 plugin-registry.json 中 source 字段变更对 manifest-resolver 的影响
3. 25 分钟估算偏乐观，建议 40-50 分钟

修正建议：
- 只需修改 `plugin-loader.js` 的路径解析逻辑，统一到 `harness-build.js` 的行为
- 增加 `sourceType` 字段（`core` / `npm` / `local`）到 registry entry，而非用字符串前缀推断
- 增加 source 解析的单元测试覆盖

**WP-069(→WP-079): 插件发现与安装** — 方向正确，CLI 膨胀风险被低估

代码验证：
- `bin/tackle.js` 已有 1,623 行、12 个子命令，无模块化结构

问题：
1. 添加 `install`/`uninstall` 命令会进一步增加 CLI 文件大小
2. npm install 操作需考虑权限、网络错误、版本锁定
3. 应先重构 CLI 为子命令模块化结构（`commands/*.js`）

#### B 类 — 开发者体验

**WP-070(→WP-080): 性能监控与诊断模块** — 方向部分偏颇

代码验证：
- `logger.js` (175 行) 记录 timestamp 不记录 duration
- `event-bus.js` (196 行) 有事件历史但无耗时统计

问题：
1. 方案过度设计：timer 逻辑放入 PluginLoader 的 activate/deactivate 流程过于侵入式，建议用中间件/装饰器模式
2. 与 WP-071(诊断命令) 有隐含依赖（诊断需要消费性能数据），报告中未标注

**WP-071(→WP-081): 诊断命令** — 方向正确，实施简单

最务实的 WP 之一。建议增加安装模式（全局 vs 本地）诊断。

**WP-072(→WP-082): 插件模板/脚手架** — 方向正确

同样面临 CLI 膨胀问题。建议模板放在独立 `templates/` 目录。

#### C 类 — 质量保障

**WP-079(→WP-083): Skill 插件结构性测试** — 方向正确，优先级应提高

代码验证：确认 15 个 Skill 插件无任何测试。

问题：报告列为 P1，但 Skill 是用户最直接接触的插件类型，建议提升为 Phase 1。

**WP-080(→WP-084): E2E 工作流测试** — 方向正确

使用临时目录做 E2E 是标准做法。25 分钟估算偏乐观（环境清理和隔离是主要消耗）。

**WP-084(→WP-088): 贡献指南更新** — 方向正确但时机不当

代码验证：CONTRIBUTING.md L89 确认 "暂未配置自动化测试框架" 过时。

问题：放在 Phase 1 时机不当（外部插件 API 未稳定），应移至 Phase 3 末尾。

#### D 类 — 工具链与文档

**WP-073(→WP-085): API 参考文档** — 方向正确

问题：报告说"从 JSDoc 生成"，但未评估 JSDoc 覆盖率。如不完整需先补全。

**WP-074(→WP-086): 插件版本兼容性检查** — 方向正确

代码验证：plugin.json 有 version 字段但无兼容性检查逻辑。

**WP-075(→WP-087): TypeScript 类型定义** — 方向正确

代码验证：确认无 `types/` 目录。API 表面约 50-60 个方法/属性，20 分钟估算偏乐观。

**WP-076(→WP-088): 多环境配置支持** — 价值存疑

代码验证：ConfigManager (307 行) 已支持 `HARNESS_<KEY>` 环境变量 + 三层配置。

问题：tackle-harness 是 CLI 构建工具，不是常驻服务。dev/staging/prod 环境配置切换在构建工具中不常见。项目级差异已被 manifest 系统覆盖。建议降级或移除。

**WP-077(→WP-089): 配置热重载** — 建议移除

问题：CLI 一次性执行模式，`fs.watch` 热重载几乎无实际价值。配置变更在下次 `tackle build` 时自动生效。建议移除此 WP。

**WP-081(→WP-090): 性能基准测试** — 方向正确

报告未标注对 WP-070(性能监控) 的依赖。

**WP-082(→WP-091): 插件发布工具** — 方向正确

依赖链正确（WP-078 → WP-079 → WP-091）。

**WP-083(→WP-092): 完整示例项目** — 方向正确但应简化

代码验证：`examples/minimal/` 已存在。建议与 WP-082(模板生成) 协同，示例可用 generate 命令生成。

**WP-085(→WP-093): Changelog 自动生成** — 方向正确

代码验证：CHANGELOG.md 已存在且维护良好，`.github/workflows/publish.yml` 已存在。

#### E 类 — 企业级特性

**WP-078(→WP-094): 插件权限与安全模型** — 方向正确但时机过早

问题：
1. 18 分最高复杂度，50 分钟估算严重偏低，实际 90-120 分钟
2. 插件运行在 Claude Code 进程中，Claude Code 本身已有工具权限控制，两者如何交互未说明
3. 过早优化：当前用户更需要稳定性而非安全隔离

建议：Phase 4 仅做设计研究，实际实现推迟到有明确企业用户需求。

**WP-086(→WP-095): 远程 Agent 设计研究** — 方向正确

仅做设计研究，复杂度合理。EventBus 的 pub/sub 模式可扩展为远程传输。

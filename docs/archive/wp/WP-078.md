## 分析范围

### 路线图原 WP-096（调整后 WP-085）: CLI 模块化重构

**验证重点**:
- `bin/tackle.js` 实际为 1,623 行单文件（已确认）
- 命令路由位于 switch 语句（line ~1579），cmdBuild/cmdValidate 等函数内联
- 评估 commands/*.js 模式是否为最优解，是否应考虑更轻量的方案（如对象映射）
- 分析拆分后对 `node bin/tackle.js` 调用路径的影响
- 40min 时间估算是否包含集成测试验证？

**分析要点**:
- switch-case 路由模式的重构方案对比（对象映射 vs 命令类 vs 文件约定）
- 共享状态（如 config, logger）如何跨命令模块传递
- 渐进式拆分 vs 一次性拆分的风险对比

### 路线图原 WP-097（调整后 WP-086）: Plugin Loader 路径解析统一

**验证重点**:
- `plugin-loader.js._loadPlugin()`（line 425）硬编码 `core/` 子目录拼接
- `harness-build.js._resolvePluginDir()`（line 399）支持绝对路径 + 相对路径
- 两者差异不仅是代码位置，逻辑行为也不同
- 15min 估算是否现实？

**分析要点**:
- 提取 `resolve-plugin-path.js` 的接口设计
- 是否需要引入 sourceType 概念（与 WP-078/原 WP-078 耦合）
- 统一后对现有 core 插件加载的回归风险

### 路线图原 WP-078（调整后 WP-082）: 外部插件加载机制

**验证重点**:
- Registry 当前格式：`{ name, source, enabled, config }`，无 sourceType
- `manifest-resolver.js`（269 行，16 方法）需扩展扫描逻辑
- `require.resolve(packageName)` 对 npm 包的可靠性
- sourceType 枚举方案 vs 字符串前缀推断（后者已被审查报告否决）
- 45min 时间估算是否充分？

**分析要点**:
- npm 包的 require.resolve 在 monorepo / workspace 场景下的行为
- 外部插件的 skill.md / index.js 格式约定
- 错误处理策略：插件加载失败时的降级方案

### 路线图原 WP-079（调整后 WP-083）: 插件发现与安装

**验证重点**:
- 依赖 WP-085（CLI 模块化）和 WP-082（外部插件加载）
- `install`/`uninstall`/`search` 三个命令的复杂度
- npm API 调用（search, install, uninstall）的可靠性
- 依赖检查（阻止卸载被依赖的插件）的实现复杂度
- **50min 估算可能严重低估** — 仅 install 命令就需要：npm install + registry 更新 + manifest 重新生成 + 验证 + 错误处理

**分析要点**:
- npm 程序化 API vs 子进程调用 `npm install` 的取舍
- registry 文件的并发写入保护
- search 命令的 npm registry API 使用和缓存策略

### 路线图原 WP-080（调整后 WP-084）: Skill 插件结构性测试

**验证重点**:
- 15 个 Skill 插件，位于 `plugins/core/skill-*`
- 每个需验证：plugin.json 必填字段、skill.md 存在非空、skill.md 关键 Section
- 批量验证方案合理，20min 估算基本准确
- 是否应扩展为所有 4 种插件类型的结构性测试？

**分析要点**:
- 测试文件组织方式（单文件批量 vs 按插件类型分）
- CI 集成策略
- 与现有 214 个测试的整合方式

## 职责

验证最终设计文档的完整性、一致性和正确性。

## 任务清单

### Step 1: 完整性验证

- [ ] 检查两份报告的核心发现是否全部被覆盖
  - WP-109: 8 维度成熟度、18 项技术债务、8 项风险、4 阶段路线图评估、10 项行动建议
  - WP-110: 6 维度架构评估、3 模块零测试、Provider 依赖链、Manifest 缺失、10 项风险、6 项建议
- [ ] 检查原始 roadmap 的关键章节是否被综合分析
- [ ] 检查文档结构完整性（有目录、有结论、有验收标准）

### Step 2: 一致性验证

- [ ] 检查文档内部无矛盾（如时间估算不自相矛盾）
- [ ] 检查与原始报告的引用一致（数据、结论无曲解）
- [ ] 检查架构方案与现有代码的可行性（模块划分合理）
- [ ] 检查两报告的分歧点是否有明确裁决

### Step 3: 正确性验证

- [ ] 验证时间估算的合理性（与两报告估算对比）
- [ ] 验证架构方案的技术可行性
- [ ] 验证安全模型的可行性（Worker Threads 与 PluginContext DI 兼容性）
- [ ] 验证风险矩阵的分级合理性

### Step 4: 输出验证报告

- [ ] 记录发现的问题（如有）
- [ ] 给出通过/不通过判定
- [ ] 不通过时给出具体修改建议

- [ ] 两份报告的核心发现覆盖率 ≥ 95%
- [ ] 文档内部无矛盾
- [ ] 与原始报告引用一致
- [ ] 时间估算在合理范围（与两报告偏差 < 20%）

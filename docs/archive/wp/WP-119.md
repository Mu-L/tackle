### 问题分析

当前所有 API（公共方法、内部方法、实验性方法）在文档和代码层面无区分。这导致：

1. 第三方开发者无法判断哪些 API 可安全依赖
2. 内部实现细节可能被外部依赖，导致破坏性变更
3. 实验性 API 无风险提示，可能被误用于生产环境
4. 23 个插件的 API 表面缺乏一致性检查

WP-113 模块化完成后，拆分出的模块接口更清晰，是进行 API 分类标注的最佳时机。

## 目标

为所有公共 API 添加 JSDoc 稳定性标注，建立 API 分类体系：

1. **@public** — 稳定的公共 API，破坏性变更遵循弃用策略
2. **@internal** — 内部 API，不保证跨版本兼容
3. **@experimental** — 实验性 API，随时可能变更

覆盖范围:
- `plugins/runtime/*.js` 所有模块的导出方法
- `plugins/contracts/*.js` 契约接口
- 23 个核心插件的 `plugin.json` 公共配置

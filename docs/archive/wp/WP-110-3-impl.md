## 分析结论

> 分析人: architect | 日期: 2026-05-29 | 基于代码事实

### 维度 1: 外部插件安全风险 — 判断: **有条件可行（需立即加固）**

**现状事实**:

1. `resolve-plugin-path.js`（161 行）支持 `core`/`npm`/`local` 三种 sourceType，通过 `require.resolve()` 加载 npm 插件，通过路径解析加载 local 插件。无任何安全校验。
2. `plugin-loader.js:462` 直接执行 `require(indexJsPath)` 加载任意 JS 模块，外部插件因此获得完全的 Node.js 进程权限（文件系统、网络、子进程等）。
3. `hook-skill-gate/index.js` 的安全措施（1MB 限制 + 原型污染防护）仅保护 stdin JSON 解析场景，不保护插件加载阶段。这是两个完全不同的攻击面。
4. `plugin-registry.json` 的 23 个条目全部是 `sourceType === "core"`（隐式默认），无 `sourceType` 字段声明，无 `capabilities` 声明，无权限模型。

**风险评估**:

| 风险 | 严重程度 | 说明 |
|------|----------|------|
| 外部插件执行任意代码 | **严重** | `require(indexJsPath)` 无沙箱，插件可执行 `child_process.exec()`、读取文件系统、发起网络请求 |
| npm 供应链攻击 | **高** | `resolveNpmPath()` 直接调用 `require.resolve(source)`，恶意 npm 包可劫持加载链 |
| 注册表注入 | **中** | `plugin-registry.json` 无签名验证，项目级 manifest 可注入恶意条目 |
| 原型污染通过 plugin.json | **低** | `_readPluginJson()` 使用 `JSON.parse()` 后直接赋值，但仅限元数据，不涉及类实例化 |

**结论**: 当前安全状态适用于 L0 封闭生态（所有插件由核心团队控制）。一旦进入 L1 开放加载，现有措施完全不足。需要在 L0→L1 过渡前至少实现 capabilities 声明和最小权限默认策略。

### 维度 2: 沙箱隔离技术评估 — 判断: **有条件可行（方案受限）**

**三种技术方案对比**:

| 方案 | 隔离强度 | 性能开销 | 零外部依赖兼容 | 实现复杂度 |
|------|----------|----------|----------------|------------|
| **Worker Threads** | 中 | 低（<50ms 启动） | 完全兼容 | 中 |
| **VM Module** (vm2/sandbox) | 低（已知逃逸） | 极低 | 需引入 vm2 或自建 | 高且不安全 |
| **子进程** | 高 | 高（>100ms 启动） | 完全兼容 | 中 |

**详细分析**:

1. **Worker Threads** — 推荐方案
   - Node.js 内置，零外部依赖，符合项目约束
   - 插件在独立线程运行，通过 `postMessage` 通信，可限制 `workerData` 暴露范围
   - 无法完全隔离文件系统访问，但可通过 `execArgv` 限制模块加载
   - 与 `PluginContext` 的 DI 模式天然契合：`postMessage` 替代直接对象引用
   - 缺点：不支持 Skill 插件（纯 Markdown），仅 Hook/Validator/Provider 需要

2. **VM Module** — 不推荐
   - Node.js 内置 `vm` 模块的沙箱可被绕过（通过 `this.constructor.constructor('return process')()` 等已知逃逸方式）
   - `vm2` 已停止维护（2023 年 7 月），引入它违反零外部依赖策略
   - 自建安全沙箱成本极高，不适合小团队

3. **子进程** — 备选方案
   - 隔离最强（独立进程、可降权运行），但通信开销大
   - 与当前 CLI 单进程架构冲突——CLI 是一次性进程，子进程管理增加复杂度
   - 适合高安全场景（如企业级部署），对当前阶段过度工程

**结论**: Worker Threads 是平衡安全性和工程成本的最优选择。建议在阶段 II（v0.3.0）引入，作为"安全沙箱 lite"方案。不需要引入外部依赖。

### 维度 3: 插件签名与验证 — 判断: **可行（但非当前优先级）**

**必要性评估**:

当前阶段（L0→L1）不需要插件签名。理由：
- 23 个核心插件全部内置，签名无实际价值
- 外部插件通过 npm 安装时，npm registry 自身有 integrity 校验（sha512）
- `resolve-plugin-path.js` 的 npm 路径解析依赖 `require.resolve()`，已受 npm integrity 保护

**签名方案复杂度**:

| 方案 | 实现成本 | 依赖 | 适用阶段 |
|------|----------|------|----------|
| npm integrity 复用 | 低（~30min） | 无（已有） | L1 |
| SHA-256 + 公钥自签 | 中（~120min） | 需引入 crypto（内置） | L2 |
| 完整 PKI 签名链 | 高（>300min） | 需密钥管理基础设施 | L3 |

**对开发体验的影响**:

- 签名验证会增加 `tackle install` 和 `tackle build` 的耗时
- 开发模式下需支持跳过签名验证（`--skip-signature-verify`）
- 本地插件（`sourceType: "local"`）通常不需要签名

**结论**: 阶段 I/II 不需要实施插件签名。阶段 III 可实现 SHA-256 + 内置 crypto 模块的轻量签名方案，不需要引入外部依赖。npm integrity 校验在 L1 阶段已足够。

### 维度 4: 生态演进路径评估 — 判断: **可行（路径合理，门槛可控）**

**逐层评估**:

#### L0 → L1（开放加载） — 门槛: 低，技术差距: 小

- **已完成**: `resolve-plugin-path.js` 支持 npm/local/core 三种 sourceType
- **已完成**: `plugin-registry.json` 支持 `sourceType` 字段
- **未完成**: `tackle install`/`uninstall`/`search` 命令（CLI 层面）
- **未完成**: 安全审查流程（非技术，但需要建立）
- **差距估算**: 约 120-180min 工程工作（CLI 命令 + 安全加固）

#### L1 → L2（策划生态） — 门槛: 中，核心挑战: 质量关卡

- **需要**: 插件质量标准执行器（`tackle validate` 扩展）
- **需要**: `tackle-plugin-*` 命名规范强制（`plugin-loader.js` 中增加校验）
- **需要**: 自动化测试要求（CI 级别的准入检查）
- **关键**: `plugin.json` schema 形式化（当前是隐式 schema，见 `harness-build.js` 的 `_validatePlugin()` 方法）
- **差距估算**: 约 300-400min（质量标准定义 + 工具链 + 首批外部插件引入）

#### L2 → L3（自治生态） — 门槛: 高，需要社区基础

- **需要**: 插件目录/市场基础设施
- **需要**: 社区治理机制（贡献者体系、review 流程）
- **需要**: 版本兼容性矩阵（`minHarnessVersion` 字段 + 运行时检查）
- **关键**: 社区规模——至少需要 10+ 活跃插件开发者才能形成可持续生态
- **差距估算**: >1000min，且依赖社区增长

**结论**: L0→L1 和 L1→L2 的技术门槛可控，路径合理。L2→L3 的门槛主要不是技术性的而是组织性的——需要社区规模支撑。Roadmap 将 L3 定位为 v1.0.0 愿景是合理的，不应过早推进。

### 维度 5: 治理机制可行性 — 判断: **有条件可行（需简化流程）**

**RFC 流程适用性**:

- **小团队场景**: 当前项目 50 次提交、1 位核心贡献者。完整 RFC 流程（提案 → 评论 → 批准 → 实施）在此规模下是过度治理。
- **简化方案**: 建议采用 "轻量 RFC" 模式——破坏性变更需要在 `docs/rfcs/` 目录下创建 Markdown 文件描述变更内容、影响范围和迁移路径，但不强制评论/审批流程。
- **自动化执法**: 可通过 CI 检测 `plugin-interface.js` 的公共 API 变更，自动触发 RFC 要求。

**API 版本化成本**:

- **技术实现**: 在 `plugin.json` 中增加 `apiVersion` 字段，`plugin-loader.js` 加载时校验兼容性。实现成本约 60min。
- **向后兼容测试**: 需要建立 API 快照测试（比较 `Plugin` 基类的方法签名）。实现成本约 120min。
- **弃用周期管理**: 需要维护 `deprecated-apis.json` 记录弃用时间表。纯文档性工作，成本低。

**与快速迭代的平衡**:

- 当前版本节奏平均 2 天一个版本，太频繁以致于 2 版本弃用周期仅 4 天
- 建议在 v0.2.0 后切换到更稳定的发布节奏（1-2 周一个小版本），使弃用周期有实际意义

**结论**: 治理机制不应照搬大型项目模式。轻量 RFC + 自动化 API 变更检测 + 稳定发布节奏是适合当前团队规模的方案。

### 维度 6: 插件市场可行性 — 判断: **可行（复用 npm registry 是最优路径）**

**技术方案对比**:

| 方案 | 维护成本 | 发现体验 | 实现复杂度 | 零依赖兼容 |
|------|----------|----------|------------|------------|
| **复用 npm registry** | 低 | 依赖 npm search | 低 | 完全兼容 |
| **自建 registry** | 高 | 可定制 | 高 | 需外部存储 |
| **静态目录（GitHub）** | 低 | 有限 | 极低 | 完全兼容 |

**推荐方案: npm registry 复用**

- `tackle search` 命令可通过 `npm search tackle-plugin-` 实现（命名约定过滤）
- `tackle install` 映射到 `npm install` + registry 条目注册
- 插件质量通过 `tackle validate` 在安装后验证
- 无需自建基础设施，零额外维护成本
- 与 `resolve-plugin-path.js` 的 `sourceType: "npm"` 已天然对齐

**与现有命令的集成**:

- `resolve-plugin-path.js:102-127` 的 `resolveNpmPath()` 已实现 npm 包解析逻辑
- 缺失的是 CLI 层命令（`tackle install`/`uninstall`/`search`）和 registry 条目的自动注册
- `manifest-resolver.js` 已支持项目级覆盖，`updatePluginInManifest()` 可用于安装/卸载流程

**维护成本估算**: 复用 npm registry 方案的增量维护成本约每月 30min（处理 npm API 变更）。自建方案则需每月 >4h。

**结论**: 复用 npm registry 是唯一合理的路径。自建 registry 对当前项目规模是不现实的投入。静态目录方案可作为过渡（v0.2.0），npm 集成在 v0.3.0 完善。

### 安全与生态建设风险等级评估

| 阶段 | 安全风险 | 生态风险 | 综合评估 |
|------|----------|----------|----------|
| **L0→L1** (v0.2.0) | **高** — 开放加载无安全模型是最大风险 | **低** — 技术差距小 | **需要安全先行** — 安全模型必须与 install 命令同步交付 |
| **L1→L2** (v0.3.0) | **中** — 沙箱隔离可降低到可接受水平 | **中** — 质量标准需要社区认同 | **可行但有条件** — 取决于 L1 阶段安全模型的有效性 |
| **L2→L3** (v1.0.0) | **低** — 签名机制 + 沙箱形成纵深防御 | **高** — 社区规模不足是最大障碍 | **长期愿景** — 技术可行，但需要社区自然增长 |

**总体判断**: Roadmap 的安全与生态方案**有条件可行**。关键条件是：

1. **L0→L1 过渡必须安全先行** — 不能在没有安全模型的情况下发布 `tackle install` 命令
2. **Worker Threads 沙箱是技术可行且零外部依赖的最优解** — 不需要引入外部库
3. **npm registry 复用是生态基础设施的唯一务实选择** — 自建 registry 成本不可接受
4. **治理机制需要轻量化** — 完整 RFC 流程对当前团队规模过度，自动化检测 + 文档记录足够
5. **L2→L3 的主要瓶颈是社区而非技术** — 不应过早投入市场基础设施建设
